# 3. Detailed Example Embodiments

***

These example flows show how SafeGen operates across different actors.\
All scenarios follow the same principle:

**No IDs. No biometrics. No personal data shared.**\
**Only a cryptographically signed “minor: yes/no” signal.**

***

## **3.1 Parent Registers Child Number**

**Step 1 — Parent verifies their own number**\
Parent enters their mobile number. SafeGen sends a one-time PIN.\
Parent enters the PIN.\
→ Parent is now a “trusted adult controller.”

**Step 2 — Parent links the child’s device**\
Parent enters the child’s mobile number.\
SafeGen sends a one-time PIN to the child’s phone.\
Parent physically accesses the child’s device and enters the PIN.\
→ Confirms legitimate parental control.

**Step 3 — SafeGen scrambles everything**\
SafeGen immediately:

* Hashes the child’s number with a telco-grade salt
* Hashes month/year-of-birth for auto-delisting
* Stores only irreversible hashes
* Deletes all raw values and does not persist raw values beyond hashing.

**Outcome:**\
The child is added to the Safelist, with **no raw phone numbers or DOBs stored — only salted hashes**.

***

## **3.2 Telco Creates Verified Attribute**

If a telco participates, the process is even simpler and population-wide.

**Step 1 — Telco gains customer consent**\
Via SMS/app, the telco asks parents to allow SafeGen inclusion.

**Step 2 — Telco creates anonymous, signed statements**\
Telco never exports subscriber data.\
Instead, it generates:

* A hashed and salted representation of each minor’s mobile number
* A signed one-bit age attribute: _“minor = true”_

**Step 3 — SafeGen receives only scrambled entries**\
SafeGen imports:

* The telco’s hashed value
* The telco’s signature

**Outcome:**\
SafeGen gains accurate nationwide minors data **without any customer information ever leaving the telco**.

***

## **3.3 SafeGen Generates Clean Knowledge Event**

SafeGen combines parent-added or telco-added safelist entries into a unified enforcement dataset.

**SafeGen processes:**

* Only hashed identifiers
* Only salted, irreversible values
* Only one-bit telco age attributes (when available)

SafeGen then creates the **Clean Knowledge Event (CKE)**:

> **A cryptographically signed answer:**\
> &#xNAN;**“This hashed number belongs to a minor.”**

Properties of a CKE:

* One bit only
* No identity
* No subscriber metadata
* No age, DOB, or account details
* Non-linkable across institutions. It is hard to link across institutions because each uses different salts and integration keys.
* Fully auditable

***

## **3.4 Platform Receives Enforcement Signal**

Platforms (social media, adult sites, app stores, etc.) use a simple query model.

**Step 1 — Platform hashes a phone number**\
They hash the user-supplied number with the same public hashing spec.

**Step 2 — Platform submits the hash to SafeGen**\
A single API call.

**Step 3 — SafeGen returns an enforcement signal**

* **TRUE** → minor
* **FALSE** → not in safelist

**Step 4 — Platform enforces law**\
Examples:

* Block under-16 sign-up
* Deny adult app installation
* Deny access to adult content
* Limit social features for minors

**Outcome:**\
Platforms get reliable enforcement with **zero identity exposure**.

***

## **3.5 Real-World Enforcement Flow**

This illustrates how all actors interact in practice.

**Scenario — A 14-year-old tries to access a restricted service**

1. Child attempts login on a social platform.
2. Platform hashes their phone number.
3. Platform queries SafeGen.
4. SafeGen returns **“minor = YES.”**
5. Platform enforces under-age restrictions.
6. Child is directed to age-appropriate pathways.
7. No identity, DOB, or personal information is exchanged **between telcos, platforms, and SafeGen — only hashes and one-bit signals**.

**Scenario — A telco-ISP enforces network blocking**

1. Device connects to the mobile network.
2. Telco-ISP recognises the SIM as belonging to a safelisted hash.
3. ISP blocks known adult domains.
4. No DPI, no packet inspection, no identity leakage.

**Scenario — App store checks eligibility**

1. User requests an 18+ app.
2. App store hashes the number provided for SMS validation.
3. Queries SafeGen.
4. If “minor = YES,” installation is denied.

Everything hinges on the **single Clean Knowledge Event**, not personal data.

***

## **3.6 Regulator Audit Flow**

Regulators need proof that:

* Minors are being protected
* Platforms acted on verified signals
* No personal data was shared

SafeGen provides an auditable model with **no access to identities.**

**Audit Components:**

**1. Signed CKE Records**\
Regulators can inspect:

* The cryptographic signatures
* Timestamped logs
* Proof that a platform received valid signals

No child data is ever visible.

**2. Platform Enforcement Logs**\
Platforms log:

* “SafeGen flagged this hashed identifier as a minor”
* “We blocked or restricted access”

Again: hashed identifiers only.

**3. Telco Assurance Evidence (optional)**\
Telcos can prove they generated:

* Signed one-bit age attributes
* Without exporting any subscriber information

**4. End-to-End Integrity Checks**\
Regulators confirm:

* SafeGen never handled raw identifiers
* Platforms never received identifying data
* No boundary crossed with PII

**Outcome:**\
Regulators achieve full oversight with **zero exposure** of personal, identifying, or sensitive data.
